# Reflection — Extracurricular Assignment

## Activity description (expectations + what I did)
For this extracurricular activity, my goal was to build a full‑stack market intelligence and trading dashboard as a portfolio project aligned with my career goals in software engineering (fintech / real‑time data systems). The expectations I set for myself were: (1) design a clear UI for an “operational” dashboard and an “analysis” view, (2) implement a backend that can serve real market data and stream live snapshots, (3) add basic automated trading logic with risk controls, and (4) document the work in a GitHub repo with proof and a reflection. I built a redesigned frontend with Dashboard, Analyzer, Markets, and Test Lab pages; implemented a live market feed pipeline using WebSockets plus a market discovery API; added an Analyzer view with probabilities and performance widgets; and fixed build issues that caused blank pages and missing CSS. I also created documentation placeholders for reflection and proof.

## Technical decisions
I separated the app into a backend and a React frontend to keep responsibilities clear (data access/streaming vs. UI). For live updates, I used WebSockets to stream a market snapshot (bid/ask, mid price, spread) because polling is slower and wastes requests for high‑frequency data. For the trading logic I focused on volatility/spread‑widening triggers and added guardrails: position sizing as a percent of balance, stop‑loss, take‑profit, and max daily loss limits. On the UI, I chose a “command center” layout (KPIs, charts, tables, and a settings sidebar) so operational state is visible at a glance. I also hardened caching/polling behavior to reduce flicker and unnecessary load.

## Contributions
This was primarily an individual project. I designed the UI structure and navigation, implemented the live market feed + discovery integration, built the Dashboard/Analyzer/Test Lab pages, fixed the build/CSS issues, and improved robustness around polling and caching. ## Quality assessment (what went well + what I’d change)
I participated consistently and made steady progress toward an end‑to‑end prototype that demonstrates real‑time data flow, risk controls, and a usable analytics UI. The strongest part is the cohesive product feel: clear navigation, a functional dashboard, and a streaming market feed. The main gaps are depth and validation: some analyzer metrics are still partially mocked, paper trading could be more tightly wired to live market events, and automated tests are limited. If I could redo the activity, I would start with an integration-test plan early (API + WebSocket + UI smoke tests), wire all analyzer metrics directly to backend analytics, and add a short demo script plus screenshots in a dedicated `proof/` folder to make grading and verification faster.
